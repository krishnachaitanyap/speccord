Getting started

Overview

speccord treats the spec as the executable contract your code is continuously checked against.

speccord is a spec-driven development CLI that works in both directions and keeps code and spec in sync:

  • Extract (code → spec) — discover the as-is contract of an existing service: API, data, events, security.
  • Generate (spec → code) — for new work: constitution → spec → plan → tasks → story → implementation, with role personas.
  • Enforce — a lifecycle state machine with entry gates, a CI drift gate, and runtime conformance that fail the build when code and spec disagree.

Hybrid by construction

Facts and pass/fail are deterministic — parsers, the contract-surface diff, the lifecycle state machine, and the gates are all plain code. Only prose is model-written, always grounded in facts the tool already established, so the model never invents an endpoint, table, or scope.

Note
Everything runs without an API key — prose just becomes deterministic skeletons. The guarantees never depend on a model.

Why speccord

  • Stack-agnostic. Discovery reads contract artifacts that exist in any language — OpenAPI, GraphQL, Protobuf, AsyncAPI, Prisma, SQL — and enterprises can declare custom providers for proprietary frameworks.
  • AI-agent native. A built-in MCP server exposes speccord as tools, resources, and prompts; one command wires it into Claude Code, Cursor, and Copilot.
  • One knob. A single scale (0–4) decides how much process you get; everything derives from it and is overridable.

Getting started

Installation

Node.js ≥ 18. No other runtime required.

Install

git clone https://github.com/krishnachaitanyap/speccord.git
cd speccord
npm install
npm run build
npm link            # puts `speccord` on your PATH (or call ./bin/run.js)

Optional: enable the model

Set an API key to get drafted prose and agent personas. Without it, every command still runs and emits deterministic skeletons.

export ANTHROPIC_API_KEY=sk-ant-...
export SPECCORD_MODEL=claude-sonnet-4-6   # optional; this is the default

Verify

speccord --help
speccord <command> --help

Getting started

Quickstart

Two minutes from zero to an enforced spec.

Existing service

speccord init --service orders
speccord discover        # parse the contract surface → report + baseline
speccord base draft      # confirm facts → base spec
speccord conform        # ✓ code matches the baseline

New service

speccord init --service gift --greenfield
speccord base new --intent "issue & redeem gift cards"

Enforce in CI

speccord gate --base origin/main   # contract changed but no spec updated → fail
speccord conform                   # running code drifted from baseline → fail
Tip
Run speccord capabilities any time to see what's enabled and which command each capability unlocks.

Concepts

How it works

Deterministic decisions; model-written prose. The line never moves.

The contract surface

speccord cares about the contract surface: API operations, data schema, events, and security/authz. Discovery extracts it; the base spec documents it; the gates ensure code and spec never disagree about it.

Deterministic vs. model-written

Deterministic (code)Model-written (prose only)
Parse OpenAPI/GraphQL/proto/SQL/events/securityDraft narrative around established facts
Diff the contract surface (drift)Plans, PRDs, stories, reviews
Lifecycle state machine + entry gatesNever invents endpoints/tables/scopes
Invariant
Anything that decides pass/fail is code, not a prompt. The model only writes the words.

Concepts

Scale & capabilities

Pick how much process with one number; everything derives from it.

Scale levels

ScaleAdds
0 prototypeconformance only — no ceremony
1 small+ plan, stories, lifecycle, CI gate
2 medium (default)+ solutioning, PRD, QA review
3 large+ analysis (brief), UX
4 enterprise+ product ownership, compliance gates

Capabilities

Each capability is an independent on/off that the scale sets a default for. Override any in speccord.config.yaml:

capabilities:
  ux: true              # turn one on regardless of scale

Packs

A pack bundles a scale + gate policy: service-brownfield, service-greenfield, product, enterprise, prototype.

speccord init --pack product
speccord capabilities --packs

Concepts

Discovery

A provider registry that's stack-agnostic — and extensible for enterprises.

Discovery detects your stack and runs the applicable providers, merging their facts into one report. Built-ins: openapi, graphql, protobuf, asyncapi, prisma, sql-migrations, sql-ddl, kafka, spring-security. Each activates only when its file type is present.

Custom providers (no fork)

Teach speccord a proprietary framework with regex/glob rules under discovery::

discovery:
  disable: [kafka]
  custom:
    - name: acme-rpc
      kind: api
      files: ["**/*.acme"]
      operations: { match: "rpc\\s+(\\S+)\\s+(\\S+)", methodGroup: 1, pathGroup: 2 }
      tables:     { match: "entity\\s+(\\w+)", nameGroup: 1 }
      scopes:     { match: "scope\\s+(\\S+)", group: 1 }
  plugins: ["./speccord-providers/internal.js"]   # DiscoveryProvider[] for complex cases
Note
Discovery facts are confirmed by a human in base draft before they become the baseline — tools propose, you ratify, code enforces.

Concepts

Lifecycle & gates

Hard transitions with entry gates — and two CI-grade drift checks.

Spec lifecycle

Draft → In Review → Approved → In Implementation → Implemented → Superseded. You can't enter a status until its entry gate passes:

  • Approved — lint passes, base version referenced. (compliance pack: + checklist complete, no open clarifications.)
  • In Implementation — every AC-n links to a test, and a plan + tasks exist.

The two enforcement checks

  • speccord gate --base <ref> — CI drift gate: a contract-surface file changed without a spec change in the same diff fails the build.
  • speccord conform — runtime drift: re-discovers the live surface, diffs the baseline, runs configured contract checks.
Important
Both exit non-zero on drift. Treat it as "the spec and code disagree — fix one of them."

Workflows

Existing service

Derive a trustworthy spec from code, then keep it honest.

speccord init --service orders
speccord discover                 # → .speccord/discovery-report.json
speccord base draft               # confirm facts; --yes to accept; --no-llm to skip prose
speccord conform                   # CONFORMANT

Change an endpoint and re-run speccord conform — it reports the drift and exits non-zero until you update the spec. From here, every change goes through the feature workflow.

Workflows

New service

Design the contract first, then implement against it.

speccord init --service giftcards --greenfield
speccord base new --intent "Issue and redeem gift cards; owns balances"
speccord constitution            # author principles

The fact tables start empty and fill in as features land. Once code exists, establish the real baseline:

speccord discover
speccord conform --update-baseline   # accept the now-real surface

Workflows

Feature workflow

The forward chain, identical in every mode.

speccord feature new --id SPEC-142 --title "Order cancellation"
speccord clarify   specs/features/SPEC-142-*.md   # resolve [NEEDS CLARIFICATION]
speccord plan      specs/features/SPEC-142-*.md   # → *.plan.md
speccord tasks     specs/features/SPEC-142-*.md   # → *.tasks.md
speccord analyze   specs/features/SPEC-142-*.md   # deterministic cross-check
speccord advance   specs/features/SPEC-142-*.md --to Approved
speccord implement specs/features/SPEC-142-*.md   # prompt pack, or --execute
Note
Pass the exact spec file, not a glob — a glob also matches the generated *.plan.md siblings.

Workflows

Story-driven dev

Context-engineered stories you can implement with no other doc open.

speccord story new --epic EPIC-1 --title "Issue refund on cancel"
speccord story list                # the board: id / status / epic / title
speccord story next                # the next story to pick up
speccord story advance <file> --to "In Progress"
speccord story implement <file>     # prompt pack, or --execute to drive an agent + tests

A story embeds the requirement, the contract slice, dev notes, and Given/When/Then acceptance criteria linked to tests. story implement --execute runs the agent task-by-task, runs the tests after each, and moves the story to Review when all pass.

AI agents

MCP server

speccord is the contract layer your agent plugs into.

speccord mcp     # MCP server over stdio: prints "ready on stdio"

What it exposes

  • read: get_base_spec, get_constitution, get_file, capabilities, status, stories, story_next
  • verify: analyze, lint, gate, conform, discover
  • transition: advance, story_advance, update_baseline
  • drive: next_action, story_tasks, mark_task
  • resources: speccord://base-spec, ://constitution, ://capabilities, ://story/{id}
  • prompts: implement-next-story, review-changes, fix-drift
The loop
The agent reads the spec, writes code, calls the verify tools, fixes what they report, and only then advances the lifecycle. Pass/fail stays deterministic.

AI agents

Wiring up agents

One command configures Claude Code, Cursor, and Copilot.

speccord agent-rules                 # all three
speccord agent-rules --tool cursor   # just one
speccord agent-rules --command "node ./bin/run.js"   # if speccord isn't on PATH
ToolMCP configRules file
Claude Code.mcp.jsonCLAUDE.md
Cursor.cursor/mcp.json.cursor/rules/speccord.mdc
Copilot / VS Code.vscode/mcp.json.github/copilot-instructions.md

The ruleset encodes the protocol: read the spec before coding, stay inside the contract, update the spec in the same change if the contract must change, then run the gates and fix until green.

Configuration

speccord.config.yaml, created by init.

service: orders
baseSpec: specs/base/orders.md
constitution: specs/constitution.md
prdPath: specs/prd.md

contractSurface:                 # files that ARE the contract (drive `gate`)
  - "**/openapi*.{yaml,yml,json}"
  - "**/*.{graphql,gql}"
  - "**/*.proto"
  - "**/db/migration/**/*.sql"
  - "**/schema.prisma"

methodology: { scale: 2 }        # the one knob
preset: standard                 # standard | compliance | lite

conformance:
  checkStructuralDrift: true
  checks:
    - { name: contract-tests, run: "./mvnw -q test -Dtest=*ContractTest" }

implement:
  agentCommand: "claude -p"      # receives each task prompt on stdin
  testCommand: "./mvnw -q test"

Resolution order (later wins): scale defaults → pack → preset → your explicit keys.
